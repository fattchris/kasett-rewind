We built and shipped v3.0.0 of the Molt AI multi-tenant agent orchestration platform. The platform orchestrates agent sessions across a production EKS cluster, handling up to 10,000 concurrent sessions with sticky session affinity stored in Redis. The orchestrator routes incoming requests to the least-loaded agent pod within tenant-specific namespaces, with fallback logic on pod failure.

OpenTelemetry tracing was integrated throughout, exporting spans with W3C TraceContext propagation to a dedicated trace endpoint. Custom metrics track active sessions, orchestration latency, and billing events. All spans include tenant_id, agent_id, and session_id for full observability.

The tenant onboarding API provisions new tenants end-to-end: creating database records, Kubernetes namespaces with RBAC, issuing API keys, and setting up Stripe billing with metered usage. The first tenant was successfully onboarded with a default quota of 100K API calls per month. Stripe integration includes usage alerts at 80% of plan limits and proration handling for mid-cycle upgrades.

Kubernetes security was hardened with NetworkPolicies enforcing strict tenant isolation—agent pods can only communicate with the orchestrator and message bus, with no inter-tenant traffic allowed. PodSecurityPolicy prevents privileged pods, RBAC follows least-privilege, and PodDisruptionBudgets ensure rolling updates don't take down all replicas. A full security audit confirmed the configuration meets production standards.

All five workstreams were validated with 47 passing integration tests covering orchestrator routing under load, OTEL span delivery, tenant onboarding flows, NetworkPolicy enforcement, and Stripe meter increments. The release was promoted from v3.0.0-rc1 to v3.0.0 and deployed to production.
