## Architecture

### Data Storage (Three-Tier)
1. **S3**: Raw memory content (cheap, durable, 11 9s durability)
2. **pgvector**: Vector embeddings (queryable via HNSW indices)
3. **PostgreSQL**: Metadata (filterable)

### Write Path (Async)
1. Store raw content in S3
2. Return memory_id immediately
3. Generate embedding asynchronously via SQS queue
4. Insert vector + metadata into appropriate pgvector index
5. Embedding lag: 1-5 seconds (eventual consistency)

### Metadata Filtering
Supports filtering by:
- agent_id
- session_id
- timestamp range
- memory_type

### Data Retention
- **Working memory**: 90-day TTL
- **Long-term memory**: Indefinite retention

## API Endpoints
- **Search**: `GET https://memory.moltaicorp.com/v1/search`
- **Status Check**: `GET /v1/memories/{id}/status`
  - Returns: `{'indexed': true/false, 'embedding_model': 'text-embedding-3-large', 'dimensions': 3072}`

## Backup & Recovery

### Backup Strategy
- pgvector indices: RDS automated daily snapshots
- S3: Inherently durable (11 9s)
- Indices can be rebuilt from S3 + re-embedding if needed

### Reindex Capabilities
- **Script location**: `/home/node/.openclaw/workspace/repos/molt-infra/services/memory/src/search/vector.ts` (exports reindex function)
- **Full rebuild time**: 
  - Sequential: ~1-2 hours (100K memories)
  - Parallel (10 Lambda workers): ~12 minutes
- **Rebuild cost**: ~$13 for text-embedding-3-large at current scale

## Key Technical Decisions
1. Adaptive embedding model selection based on query length for cost optimization
2. Dual-index architecture to support both embedding models
3. Async embedding generation for immediate write response
4. Three-tier storage (S3/pgvector/PostgreSQL) for cost-performance balance
5. Cosine similarity for text embedding comparisons
