The session covered five parallel infrastructure and tooling tracks, all in planning or early implementation stages. The primary focus was establishing the technical specifications and resource identifiers for each workstream.

The notifications consolidation merges two legacy services (molt-email-notifications and molt-push-notifications) into a unified molt-notifications service. The new repository is hosted at github.com/moltaicorp/molt-notifications. The migration strategy starts with email channel implementation, followed by push notifications, with both old services decommissioned once feature parity is confirmed.

Logging infrastructure is being centralized into Datadog, moving away from CloudWatch for application logs. CloudWatch will be retained only for AWS-native service logs. The Datadog API key is stored in kv/prod/datadog/api-key, with logs flowing to datadoghq.com and a 90-day retention policy.

An admin tool for the customer success team is being deployed at https://admin.moltaicorp.com. The tool provides customer lookup, plan changes, refund issuance, and audit trail capabilities, all gated behind the CustomerSuccessAdmin role.

The SDK pipeline publishes TypeScript and Python SDKs to npm (@moltai/sdk) and PyPI (moltai-sdk) respectively. GitHub Actions handles automatic publishing on git tags matching v*. Current version is 3.4.0.

A read replica for reporting workloads is being provisioned at molt-prod-ro.cluster-ro-xyz.us-east-1.rds.amazonaws.com on a db.r6g.2xlarge instance. Tableau and Looker will connect using the reporting_ro role. The instance class was deemed sufficient for current reporting load.
