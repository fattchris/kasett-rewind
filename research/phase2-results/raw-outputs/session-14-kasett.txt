The session covered five parallel infrastructure and tooling tracks, all in active planning or early implementation. The primary focus was merging two separate notification services into a unified system, alongside provisioning new infrastructure for reporting, logging, admin tooling, and SDK distribution.

The notifications consolidation merges molt-email-notifications and molt-push-notifications into a single service called molt-notifications, hosted at github.com/moltaicorp/molt-notifications. The migration strategy starts with the email channel, then adds push notifications, with both legacy services retired once feature parity is confirmed.

Logging is being centralized into Datadog, moving application logs away from CloudWatch (which will remain only for AWS-native logs). The Datadog site is datadoghq.com with 90-day retention, using an API key stored in kv/prod/datadog/api-key.

A customer success admin tool was deployed at https://admin.moltaicorp.com, providing customer lookup, plan changes, refund issuance, and audit trail capabilities, all gated behind the CustomerSuccessAdmin role.

The SDK pipeline publishes TypeScript and Python SDKs to npm:@moltai/sdk and pypi:moltai-sdk respectively, with GitHub Actions auto-publishing on git tags matching v*. Current version is 3.4.0.

A read replica for reporting workloads was provisioned at molt-prod-ro.cluster-ro-xyz.us-east-1.rds.amazonaws.com on a db.r6g.2xlarge instance, with Tableau and Looker pointed at it using the reporting_ro role.
