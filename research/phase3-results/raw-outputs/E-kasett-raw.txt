We spent the session responding to a production API incident. Latency spiked from 80ms to 2400ms at 08:42 UTC, with error rates climbing from 0.3% to 1.1%. The regression was traced to a deploy by Anna two minutes earlier — a response-serialization refactor in `src/server/serialize.ts` that accidentally introduced a synchronous DB call on every response instead of an async cached path. We decided to rollback first to restore service, then fix properly in code review without production pressure.

The rollback command (`molt-deploy rollback --service=api --to=v2.14.7`) pinned us back to last Tuesday's known-good release. Latency normalized within three minutes as new pods replaced old ones. Anna is now working on the hotfix PR, but CI is flaky — the new regression test is timing-sensitive and fails about 1 in 5 runs. She's rewriting it to use fake timers. A postmortem doc was started and an action-item review is scheduled for Friday 3pm MT, following standard incident cadence (48 hours after resolution).
